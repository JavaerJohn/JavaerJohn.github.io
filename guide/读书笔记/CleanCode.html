# 代码整洁之道

# 变量命名

1. 名副其实

通过变量、函数、类的命名即可明白其含义，若名称需要注释来补充，那么就不算是名副其实。例如在扫雷游戏中，使用变量 list 来表示游戏盘面，list 的含义比较模糊，可将其修改为 gameBoard。

2. 避免误导

避免使用专有名称，如 hp、aix、sco 等是 Unix 平台中的专有名称，代码中应避免这些变量名。此外，还应避免相似度较高的名称。

3. 做有意义的区分

变量的名称应该体现出有意义的区分，例如 Product、ProductInfo 和 ProductData 虽然名称不同，但意义却无差别，同时使用这些变量会对读者造成误导，可以考虑统一使用更为简短的 Product。

4. 变量名应可读、可搜索

可读的变量更有利于读者理解其含义，也更容易记忆，例如 genymdhms（声称日期，年、月、日、时、分、秒）就没有 generationtimestamp 容易理解。可搜索的变量更有利于定位问题，例如 RETRY_TIIMES = 7 可使用 RETRY 或者 TIMES 关键字搜索出来，而数字 7 则不容易找到。

尽量不要使用单字母名称，单字母名称仅适用于短方法中的本地变量。

5. 类名和方法名的注意事项

类名和对象名应当是名词或者名词短语，如 Customer、WikiPage 等。方法名应当是动词或者动词短语，如 deletePage、save 等。

6. 每个概念对应一个词

为每一个抽象概念选一个词，并且一以贯之。fetch、retrieve、get 都有获取的意思，若不同的方法采用不同的单词进行命名，如 fetchA 和 getB，那么调用者就需要花费较多的时间去查阅方法的使用。

7. 避免双关

避免将同一个单词用于不同的目的，例如不能将 add 既用于变量拼接/相加，又用于向集合中添加元素（可使用 insert 代替）。

8. 添加有意义的语境

大多数的名称都不能够自我说明，我们需要使用命名良好的类、函数或名称空间来放置名称，给读者提供语境。如果没有这样做，那么就得给名称添加前缀。例如 street、houseNumber、city、state 和 zipcode 组合在一起就构成了一个地址，但如果在方法中看到孤零零的一个 state，那么该变量并不容易推断出 为某个地址的一部分。解决方法有两种，一种是使用 Address 类封装这些变量，通过含义明确的 Address 来为读者提供语境；另一种是添加前缀，如 addrState。

9. 删除没用的语境

语境太过充分未必是一件好事，因为这会导致变量名过长，通常只要短名称足够清楚，那么该名称就比长名称好。例如有一个名为 CUPT 应用，在其中给每一个变量都增加一个 CUPT 的前缀就不是什么好点子，某些变量（如 address）本身已经足够明确，不需要额外添加前缀/后缀。
